# Changelog

This file is maintained **by hand** — record notable changes in the
`[Unreleased]` section as part of the PR that makes them. Format follows
[Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

Per-version release notes for tagged releases are published on the
[GitHub Releases page](https://github.com/wyre-technology/conduit/releases) —
`semantic-release` generates them from commit history at release time. It no
longer commits version sections back into this file (that required a direct
push to `main`, which branch protection correctly rejects); the released-version
detail lives on Releases, the human narrative lives here.

The version history below (1.11.0 and earlier) was generated by
`@semantic-release/changelog` before the switch and is kept as the record.

## [Unreleased]

### Added

- **conduit-prod vendor fleet — `azure/vendor-fleet.bicep` (option-a Piece 1).**
  Provisions the conduit-prod vendor MCP sidecar fleet — one internal-ingress
  Container App `<slug>-mcp` per vendor — into the existing `conduit-prod-env`
  managed environment. A separate template from `main.bicep`: the gateway
  template still does not own the fleet; this one owns the conduit-prod fleet,
  so a gateway redeploy stays lean. The fleet is the **33 internal-sidecar
  vendors** of `vendor-config.ts` (the entries whose `containerUrl` is
  `http://<slug>-mcp:8080`; the 8 external-hosted vendors need no container).
  `azure/vendor-fleet.conduit-prod.bicepparam` carries the 33-vendor list,
  each image **digest-pinned** (resolved from `ghcr.io/wyre-technology/
  <slug>-mcp:latest`). Naming the apps `<slug>-mcp` lets the gateway resolve
  every vendor via the `vendor-config.ts` default `containerUrl` with no
  per-vendor `VENDOR_URL_*` wiring. `azure/vendor-fleet-completeness-check.sh`
  is the acceptance gate — it parses `vendor-config.ts` for the sidecar set
  and asserts each `<slug>-mcp` app exists healthy in `conduit-prod-env`, so a
  missing vendor is caught at deploy time rather than as a customer-facing
  "Tool discovery failed: fetch failed" on the Tool Access page.
- **`conduit-prod` deploy path in `deploy.yml` (Phase B).** `deploy.yml` gains
  a third `workflow_dispatch` target — `conduit-prod` — that deploys the
  dedicated own-RG conduit-prod stack (`rg-conduit-prod`, World-A separation).
  Unlike staging/production it keeps all secrets (the 4 deploy params +
  `rootlyWebhookUrl`) in its own `conduit-prod-kv` and fetches them at deploy
  time (the deploy SP holds `Key Vault Secrets User` there) — no
  GitHub-environment secrets to duplicate. The step also runs an idempotent
  `az postgres flexible-server parameter set azure.extensions=PG_TRGM` before
  the deploy, so a fresh conduit Postgres never crashloops `migrate.ts` on the
  unallow-listed `pg_trgm` extension (migration 012). The `Read existing
  gateway state` step is env-resolved for conduit-prod's `rg-conduit-prod` /
  `conduit-prod-gateway` naming. `azure/CONDUIT-PROD-DEPLOY.md` documents the
  `pg_trgm` allow-list and the gateway-identity → Key Vault grant as the two
  fresh-stack standup hardening steps.
- **Track A — Audit Log tab wired to real data.** The reseller console's
  per-customer Audit Log tab (`/org/customers/:id/audit`) now serves a live
  reseller-scoped feed of the customer org's MCP tool invocations instead of
  mock rows. New endpoint `GET /admin/reseller/:resellerId/customers/
  :customerId/audit` (in `src/reseller/routes.ts`), gated by
  `requireResellerOrCustomerAccess` — the same `:id`-ownership gate the live
  Usage tab uses, with `request_log` RLS underneath (enforced twice). The web
  page additionally swaps the bare `requireResellerAccess` for an
  `:id`-ownership check before rendering any real customer identity, so a
  reseller cannot open `/audit` for a customer it does not own. The tab is a
  live client-fetch shell (`createElement`/`textContent`, no `innerHTML`).
  First of the six mock-first per-org tabs to go live; v1 surfaces tool
  invocations (admin-event entries are a documented follow-up).
- **Two Microsoft MCP vendors mirrored from the WYRE MCP gateway for parity.**
  `microsoft-graph` (Microsoft Graph Enterprise) — a hosted first-party,
  preview-tagged OAuth vendor at `mcp.svc.cloud.microsoft/enterprise`, with no
  sidecar. `azure-mcp` (Azure MCP Server) — a sidecar vendor authenticated by an
  Entra service principal (tenant ID / client ID / client secret), added to
  `docker-compose.yml` with `VENDOR_URL_AZURE_MCP` wired into the gateway. A new
  optional `preview?: boolean` field on `VendorConfig` surfaces a "Preview"
  badge without affecting proxy behavior.
- **Track C — reseller-console / subtenant UI/UX.** The customer-facing
  surfaces a reseller uses to administer its customer orgs, behind the new
  `requireResellerAccess` gate (`requireTeamAccess` + `org.type === 'reseller'`):
  - **Reseller-console shell** — `customer-detail` / `reseller-settings` nav
    modes and the reseller-console nav (Customers, Hierarchy).
  - **Surface 1 — Customers** (`/org/customers`): the customer-org list with
    search + plan filter.
  - **Surface 2 — Customer Detail** (`/org/customers/:id`): per-customer
    overview; analytics wired live to the reseller-scoped customer-dashboard
    endpoints, the rest mock-data-first.
  - **Surface 3 — Onboard MCP wizard** (`/org/customers/:id/onboard-mcp`):
    4-step wizard to onboard an MCP integration for a customer.
  - **Surface 4 — Nested Hierarchy** (`/org/hierarchy`): the reseller →
    customer → subtenant tree, a vertical collapsible view.
  - **Surface 5 — White-Label Branding** (`/org/reseller/branding`) plus the
    reseller-settings stub nav (General / Billing & Plans / API & Webhooks /
    Audit Log).
  - **Sub-customer onboarding wizard** (`/org/customers/new`): 3-step
    provisioning flow for a new customer org.
  - **Tenant switcher**: a sidebar dropdown in the customer-detail context to
    move between the reseller and its customers.
  - **Per-org management tabs** (`/org/customers/:id/{mcps,users,usage,tools,
    audit,billing,settings}`): the seven customer-detail tabs a reseller uses
    to administer one customer org. Usage loads live from the reseller-scoped
    customer-dashboard endpoint (`:id`-ownership-checked); the other six are
    mock-data-first behind documented Track A swap-in seams.
  All surfaces are mock-data-first except the Surface 2 analytics; consequential
  actions (create customer, onboard) render disabled behind documented Track A
  swap-in seams.
- **`errorRate` on `UsageSummary` + Error Rate dashboard stat card.**
  `getUsageSummary()` now computes `errorRate` — the fraction (0–1) of
  `request_log` rows in the org/date window whose HTTP `status_code` is an
  error (`>= 400`), `0` when there is no traffic (`NULLIF` guards the
  divide-by-zero). The operator team dashboard gains an "Error Rate" stat
  card. The Track C S2 customer-detail frame's Error Rate (7d) card reads the
  same field. Regression guard:
  `src/dashboard/usage-summary-error-rate.integration.test.ts`.

### Removed

- **Dead `MICROSOFT_CLIENT_ID` / `MICROSOFT_CLIENT_SECRET` bicep config.**
  `gateway-app.bicep` set `MICROSOFT_CLIENT_ID`/`MICROSOFT_CLIENT_SECRET` env
  vars from `main.bicep` params. `config.ts` reads them only as fallbacks —
  `azureClientId = AZURE_AD_CLIENT_ID ?? MICROSOFT_CLIENT_ID ?? ''`,
  `azureClientSecret = AZURE_AD_CLIENT_SECRET ?? MICROSOFT_CLIENT_SECRET ?? ''`
  — and `AZURE_AD_CLIENT_ID`/`AZURE_AD_CLIENT_SECRET` are *unconditionally*
  provided by `gateway-app.bicep` (KV-backed `secretRef`s) on every deploy
  path, so the `MICROSOFT_*` fallback branch is never reached. Removed both
  env vars and both bicep params (`microsoftClientId`, `microsoftClientSecret`)
  from `main.bicep` + `gateway-app.bicep`. No deploy path set these params, so
  this is a pure dead-config removal. (`config.ts` keeps the now-unreachable
  `?? process.env.MICROSOFT_CLIENT_*` fallback as harmless dead code — a
  separate app-code cleanup, out of scope here.)

- **Interim `Install Azure CLI` step in `deploy.yml`.** The deploy job ran an
  apt-install of a version-pinned `azure-cli` because the `[self-hosted,
  wyre-private]` runner pool ran the bare `myoung34/github-runner` image, which
  ships no `az`. The pool has since rolled onto the `wyre-aca-runner` image
  (`task_1779172079827`), which bakes in the Azure CLI and Bicep, so the
  install step is dead weight — `azure/login` and `azure/arm-deploy` find `az`
  on PATH directly. Removed the step and its comment block.

- **`azure/CONDUIT-PROD-DEPLOY.md` — `conduit.wyre.ai` cutover sequence
  pre-documented.** The cutover (runbook step 9) is now expanded into the six
  precise commands that run at Aaron's launch-window greenlight — RECORD 2 add
  → delete the auto-pending cert from the pre-bind attempt → create a
  controlled-name managed cert matching the bicepparam → bind → validate. Two
  traps documented inline: (a) the pre-bind ACA auto-cert ships with a random
  `-NNNN` suffix that does NOT match the bicepparam's `managedCertName`, so
  letting it provision would fail the next bicep deploy on a customDomain
  binding referencing a nonexistent cert — delete + recreate with the
  controlled name keeps bicep-as-source-of-truth; (b) the controlled-name cert
  uses HTTP-01, which only succeeds once RECORD 2 points the domain at the
  env, so the cert resource is created strictly AFTER RECORD 2 lands. One
  controlled session, not stretched.
- **`azure/vendor-fleet-completeness-check.sh` — probe-port note.** Added a
  comment block noting that a Job-to-CA same-env MCP probe must use port 80
  (the CA's internal HTTP ingress), NOT the container's targetPort `:8080`.
  Discovered 2026-05-21 when the Piece-1 acceptance probe against `:8080`
  returned `http=000` for all 33 — a false 0/33 signal that looks like total
  fleet outage but is actually Jobs not having direct-targetPort access (only
  CA-to-CA does — the gateway's `vendor-config.ts` default
  `http://<slug>-mcp:8080` is correct for the gateway and stays).

### Fixed

- **`conduit-prod` deploy no longer fails on empty gateway-state JSON.** The
  `Read existing gateway state` step queried the existing container app's
  custom domains / `VENDOR_URL_*` vars and wrote them to temp files passed to
  the Bicep deploy via `--parameters @file`. On a fresh stack with no custom
  domains, `az`/`jq` yields an empty string, so `printf` wrote an **empty
  file** — and `az deployment group create` then failed with `Failed to parse
  string as JSON: Expecting value: line 1 column 1 (char 0)`. The step now
  defaults both values to `[]` when empty (and `jq` tolerates a null `env`
  array via `(. // [])[]`), so a fresh-stack deploy passes valid empty JSON.
- **`microsoft-graph` / `m365` vendor OAuth now has real client credentials.**
  Both vendors' `oauthConfig.clientIdEnv` reads `MICROSOFT_CLIENT_ID` /
  `MICROSOFT_CLIENT_SECRET`, but no deploy path ever set those env vars (the
  bicep params defaulted to empty string and were later removed as "dead
  config") — so `buildAuthorizeUrl` would throw `Missing environment
  variable` on any connect attempt. A multi-tenant Entra app registration
  ("Conduit Graph Enterprise MCP Connector", appId
  `7dd3420d-a28f-4cc3-83e8-29113663c7b6`) now backs both vendors; its
  client id + secret are stored as `microsoft-client-id` /
  `microsoft-client-secret` in each environment's Key Vault, and
  `gateway-app.bicep` wires them into the container app as KV-backed
  `secretRef` env vars — the same pattern `AUTH0_CLIENT_*` uses. The Entra
  app holds all 41 read-only `MCP.*` delegated scopes; each customer tenant
  still grants admin consent out of band when it connects.
- **`/org/billing` no longer shows fabricated billing data.** The billing page
  rendered a hardcoded mock payment method (`visa •••• 4242`), a mock upcoming
  invoice, four fake invoices, and a `creditsUsed` figure derived as 37% of the
  allocation. `creditsUsed` is now the real value from
  `CreditService.getUsageThisMonth(org.id)` (sum of `credit_ledger` since the
  month start). The mock payment-method / next-invoice / invoice-history
  sections are replaced by a single honest **Billing details** block: an org
  with a Stripe customer gets an *Open billing portal* button (`POST
  /api/billing/portal` → the customer's real Stripe portal, which natively
  shows their real card, invoices, and upcoming charge); an org with no Stripe
  customer gets an honest "billing is managed directly" state with no button.
  No fabricated billing data reaches the page. On-page invoice rendering
  remains deferred to the Layer 1 §8 per-seat billing work.
- **`tools/call` on the unified `/v1/mcp` endpoint hung — `Promise.all` of
  service-method calls stalled the request reserved-tx connection.** Every
  `tools/call` reached `shouldCapturePrompt`, whose
  `await Promise.all([billingGate.canUsePromptCapture(orgId), orgService.getPromptCaptureEnabled(orgId)])`
  issues two concurrent DB queries on the request's single reserved
  transaction connection — which stalls it (TLDIAG-localized: the tool-call
  handler entered `shouldCapturePrompt` and never returned). Same hang class
  as the `tools/list` bug (PR #196); the precise postgres.js cause remains
  uncharacterized (tracked post-launch). The fix is mechanism-independent —
  sequentializing such a `Promise.all` into awaited calls is correctness-
  preserving and removes the concurrency. Six sites carrying that shape on
  the reserved-tx path are sequentialized: `shouldCapturePrompt`
  (`audit/prompt-capture.ts`), the inline prompt-capture gate in the audit
  page (`web/routes.ts`), `getAllToolAllowlists` (`org/tool-allowlist-service.ts`),
  the team-credential check in `oauth/authorization-server.ts` (now a single
  set-based query), the reseller-detail route (`reseller/routes.ts`), and
  `CreditService.getTotalAvailable` / `hasCreditsRemaining`
  (`billing/credit-service.ts`). `Promise.all`s of raw `sql\`\`` query
  objects (the postgres.js-idiomatic, pipelined pattern — e.g. the dashboard
  and audit-list queries) are unaffected and left as-is. This supersedes
  PR #196's flat-vs-nested framing: the operative axis is the shape of the
  `Promise.all` members, not its arity.
- **Reseller console — subtenant UI hardening pass.** A QA sweep of the
  merged subtenant surface (reseller console, both wizards, the 7 per-org
  tabs) fixed the unambiguous defects. Security: hex colors bound into
  `style=""` contexts (branding swatches, onboard-mcp catalog icons) now
  validate via a new `safeCssColor` helper rather than HTML-escape only —
  `escapeHtml` does not neutralize a CSS context; logo `src` URLs are
  https-scheme-checked (`safeHttpsUrl`); the hierarchy tree walk carries a
  visited-set + depth cap so a cyclic or pathologically deep Track A tree
  cannot overflow the stack. Correctness: the wizard step coercers reject
  `parseInt`-salvageable garbage; an unknown customer-tab renders a neutral
  body instead of the editable-looking Settings form; Settings inputs are
  read-only until their write endpoint lands. Accessibility: ARIA tree roles
  on the hierarchy, `aria-current` on wizard steppers, `th scope`, screen-
  reader state text on tool-access checkboxes, dropped a misleading
  `role="menu"` from the tenant switcher. Plus responsive fixes (mobile
  table labels, narrow-viewport steppers) and amber/red hex → theme tokens.
- **Request-context: connection leak on client disconnect during `reserve()`,
  and unbounded `reserve()` wait on pool exhaustion.** The request-context
  plugin's `onRequest` hook registered its raw-socket-`close` listener AFTER
  `await openRequestContext`, so a client that disconnected during the
  `reserve()` acquire window left the eventually-acquired reserved connection
  with no release path — `onResponse` never fired because the response never
  completed. The listener is now registered synchronously BEFORE the await;
  pre-acquire closes set an `abortedEarly` flag that the post-await check
  honors by releasing the late-arriving handle immediately (idempotent with
  the existing settle-once guard). Separately, `postgres.js`'s `reserve()`
  has no acquire timeout, so a pool-exhausted gateway would hang every new
  request forever. `openRequestContext` now races `reserve()` against a 5s
  timer (`RESERVE_TIMEOUT_MS`); on timeout it throws a typed
  `RequestPoolBusyError` which the plugin maps to HTTP 503 — fail loud, not
  hang. The losing reserve promise is attached with `.then(r => r.release())`
  so a late acquisition does not silently strand a pool slot.
- **`tools/list` on the unified `/v1/mcp` endpoint hung indefinitely.** The
  `aggregateTools` Phase-1 vendor discovery fanned out a per-org/per-team
  `Promise.all` of DB queries onto the request's single reserved-transaction
  connection; a nested / dynamic-fan-out of concurrent queries on that
  connection deadlocks (flat-fixed concurrent `Promise.all`s on the same
  connection are unaffected — the precise postgres.js cause is tracked
  separately as a post-launch investigation). Phase 1 is now a fixed, small
  number of sequential set-based queries (`listOrgVendorsForOrgs`,
  `getUserTeamIds`, `listTeamVendorsForTeams`) — `O(1)` query count instead of
  the previous `O(orgs × teams)` N+1. Phase 2 is restructured into three
  passes: sequential credential resolution, a concurrent (DB-free) HTTP tool
  fetch, then sequential allowlist filtering — removing all concurrent DB
  queries while keeping per-vendor tool fetches parallel. The credential
  injector's team tier likewise replaces its per-team `Promise.all` fan-out
  with a single set-based `getTeamCredentialsForTeams` query.
- **Microsoft/Entra sign-in broken on staging — malformed double-scheme
  callback URL.** `params.staging.bicepparam` never set `allowedHosts`, so the
  staging gateway got `ALLOWED_HOSTS=''`. `config.ts` splits that to an empty
  allowlist; `getRequestBaseUrl()` then cannot match the request `Host` and
  falls back to its hardcoded `http://localhost:8080` literal, producing
  `redirect_uri=https://http://localhost:8080/auth/microsoft/callback` —
  Microsoft rejects it. Now sets `allowedHosts = 'staging.conduit.wyre.ai'`
  (bare host, no scheme). The production deploy already passes `allowedHosts`
  inline; staging never did.

- **Staging/production deploy failed — `azure/arm-deploy` strips quotes from
  inline JSON `--parameters`.** `existingCustomDomains` and `existingVendorEnv`
  were passed inline as `param=<json>`; `arm-deploy` strips the double quotes,
  so `[{"name":"x"}]` reached `az` as `[{name:x}]` — unparseable
  (`Failed to parse string as JSON`). The `Read existing gateway state` step now
  writes the JSON to files and the deploy steps pass them as `param=@file`,
  keeping the JSON off the command line so quotes survive. Fixes both the
  staging and production deploy steps. Surfaced once the `BCP258` fix let the
  deploy reach template validation for the first time.
- **Staging deploy targets the existing managed environment.**
  `params.staging.bicepparam` now pins `containerEnvName = 'mcpgw-staging-env-v2'`.
  Unset, `main.bicep` resolved the staging managed-environment name to the
  bicep-canonical `mcpgw-staging-env`, so a deploy would have *created a new
  managed environment* and re-homed the gateway onto it. The live staging stack
  runs on the hand-built `mcpgw-staging-env-v2`; pinning it keeps a deploy an
  in-place revision bump. Mirrors the production step's existing
  `containerEnvName=mcpgw-prod-env-v2` pin.

- **Staging deploy pipeline never worked — `BCP258` on the bicepparam.**
  `params.staging.bicepparam` and `params.conduit-prod.bicepparam` omitted the
  four `@secure` params (`masterKey`/`jwtSecret`/`pgPassword`/`ghcrToken`) on
  the false assumption that CI `--parameters` overrides would fill them. A
  `.bicepparam` file must self-assign every required (non-defaulted) param of
  its `using` target — CLI overrides can only override params it already
  assigns. Both files now assign the four via `readEnvironmentVariable()`;
  `deploy.yml`'s staging step supplies them as `env:` vars from the staging
  GitHub environment secrets, and the conduit-prod Phase-B runbook exports them
  before `az deployment`. This is why every `deploy.yml` staging run had failed.

- **Docs site never shipped in the gateway image — `/docs` 404'd.** `src/index.ts`
  serves the Astro Starlight docs site from `public/` via `@fastify/static`
  (registered only when `public/` exists, so a docs-less image boots clean),
  and the site builds with base path `/docs`. But the `Dockerfile` never built
  the docs or copied them in — `public/` is gitignored and was absent from
  every image, so `existsSync(publicDir)` was always false and `/docs` 404'd.
  CI builds the docs (the `docs-build` check, 26 pages) only to discard the
  artifact. Added a digest-pinned `docs-builder` stage to the `Dockerfile` and
  `COPY --from=docs-builder /docs/dist ./public/docs`, so the gateway image now
  bundles and serves the documentation at `/docs`.

### Added

- **`deploy.yml` guard step — reject an unset or empty staging deploy secret.**
  `readEnvironmentVariable()` throws only on an *unset* var; since CI always
  defines the `env:` vars, a missing GitHub-environment secret resolves to `''`
  and would otherwise compile a blank `@secure` param. The new "Assert staging
  deploy secrets present" step fails the deploy loudly before it runs if any of
  `MASTER_KEY`/`JWT_SECRET`/`PG_PASSWORD`/`GHCR_TOKEN` is unset or empty.
