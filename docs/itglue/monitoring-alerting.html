Vendor Health Monitoring

The gateway includes a built-in VendorMonitor service that probes every configured vendor MCP container on a regular interval.

How It Works

  1. Sends a JSON-RPC initialize request to each vendor's MCP endpoint every 60 seconds (configurable via MONITOR_INTERVAL_MS)
  2. Tracks consecutive failures per vendor
  3. After 3 consecutive failures, marks the vendor as DOWN and sends a webhook alert
  4. When a vendor recovers, sends a RECOVERED alert with downtime duration

Webhook Alerts

Configure MONITOR_WEBHOOK_URL with a Discord or Slack webhook URL. Alert messages:

EventMessage Format
Vendor goes downVendor DOWN: {slug} -- {error} after 3 consecutive failures
Vendor recoversVendor RECOVERED: {slug} -- back up after {duration}

Log Analytics Queries

Useful KQL Queries

Recent 5xx errors:

ContainerAppConsoleLogs_CL
| where ContainerAppName_s == 'mcpgw-prod-gateway'
| where Log_s has '"statusCode":5' or Log_s has '"res":{"statusCode":5'
| project TimeGenerated, Log_s
| order by TimeGenerated desc
| take 50

Auth failures (401/403):

ContainerAppConsoleLogs_CL
| where ContainerAppName_s == 'mcpgw-prod-gateway'
| where Log_s has '"statusCode":401' or Log_s has '"statusCode":403'
| summarize count() by bin(TimeGenerated, 15m)
| render timechart

Response time percentiles:

ContainerAppConsoleLogs_CL
| where ContainerAppName_s == 'mcpgw-prod-gateway'
| where Log_s has '"responseTime"'
| extend parsed = parse_json(Log_s)
| extend responseTime = todouble(parsed.responseTime)
| where isnotnull(responseTime)
| summarize p50=percentile(responseTime,50), p95=percentile(responseTime,95), p99=percentile(responseTime,99) by bin(TimeGenerated, 5m)
| render timechart

Container restarts (all apps):

ContainerAppSystemLogs_CL
| where ContainerAppName_s startswith 'mcpgw-prod-'
| where Reason_s == 'BackOff' or Reason_s == 'CrashLoopBackOff' or Reason_s has 'restart'
| summarize restarts=count() by ContainerAppName_s, bin(TimeGenerated, 1h)
| order by TimeGenerated desc

Database connection errors:

ContainerAppConsoleLogs_CL
| where ContainerAppName_s == 'mcpgw-prod-gateway'
| where Log_s has 'ECONNREFUSED' or Log_s has 'connection terminated' or Log_s has 'connect_timeout'
| project TimeGenerated, Log_s
| order by TimeGenerated desc

Log Shipping to External SIEM

The gateway supports forwarding audit logs to external SIEM platforms. This is configured per-organization via the API or web UI.

Supported Platforms

PlatformAdapterEndpoint FormatAuth
Grafana LokiLokiAdapterhttps://loki.example.com/loki/api/v1/pushBasic auth (username/password in config)
GraylogGraylogAdapterGELF HTTP input URLToken or Basic auth
LogScale (Humio)LogScaleAdapterhttps://cloud.humio.com/api/v1/ingest/jsonIngest token in config

Data Shipped

Two log sources are shipped per organization:

SourceTableContents
Request Logrequest_logEvery MCP tool call: user, vendor, tool name, status code, response time
Admin Audit Logadmin_audit_logAdmin actions: member changes, role updates, credential operations, settings

Shipping Mechanism

  • The LogShipper service runs on a background interval
  • Uses cursor-based pagination (tracks last_shipped_at per destination per source)
  • Ships in batches of 500 events
  • Errors are recorded in a ring buffer (last 50 per destination)
  • Errors are viewable via the API: GET /api/orgs/:orgId/log-shipping/:destId/errors

Configuration API

# Create a log shipping destination
POST /api/orgs/:orgId/log-shipping
{
  "label": "Production Loki",
  "platform": "loki",
  "endpointUrl": "https://loki.internal.example.com/loki/api/v1/push",
  "config": { "username": "...", "password": "..." }
}

# List destinations
GET /api/orgs/:orgId/log-shipping

# Enable/disable
PATCH /api/orgs/:orgId/log-shipping/:destId/enabled
{ "enabled": false }

# View recent shipping errors
GET /api/orgs/:orgId/log-shipping/:destId/errors

  • Token savings from CLI wrapper usage
  • Request volume over time
  • Per-user activity
