Architecture Overview

The MCP Gateway is a white-label, multi-tenant AI tool gateway for managed service providers. It sits between MCP clients (Claude Desktop, Claude Code) and vendor API MCP servers, handling OAuth 2.1 authentication, credential encryption, RBAC, audit logging, and request proxying.

Network Topology

Azure Container Apps Environment

All containers run within a single Azure Container Apps managed environment. The gateway is the only container with external ingress. All vendor MCP server containers use internal-only ingress and are not accessible from the public internet.

Docker Compose (Local Development)

The docker-compose.yml defines the full stack for local development:

  • gateway — The main API server, ports 8080:8080
  • postgres — PostgreSQL 16 Alpine, ports 5432:5432, with named volume pgdata
  • 22 vendor MCP containers — Each exposed internally on port 8080 with AUTH_MODE=gateway

DNS & Domain Configuration

The gateway uses a custom domain configured via the customDomain Bicep parameter. Azure Container Apps provides an automatic FQDN (e.g., mcpgw-prod-gateway.*.azurecontainerapps.io) plus the custom domain binding.

RecordTypeTargetPurpose
mcp.wyretechnology.com (example)CNAMEAzure Container Apps FQDNPublic gateway endpoint
Note: The BASE_URL environment variable and AUTH0_CALLBACK_URL must match the custom domain. Auth0 application settings must also list the callback URL.

SSL/TLS Certificate Management

TLS is handled by Azure Container Apps managed certificates. The Bicep template references a managed certificate by name:

customDomains: [{
  name: customDomain
  certificateId: '${containerEnv.id}/managedCertificates/${managedCertName}'
  bindingType: 'SniEnabled'
}]
AspectDetail
ProviderAzure Managed Certificates (free, auto-renewed)
BindingSNI-based TLS
RenewalAutomatic (Azure handles expiry)
Internal trafficHTTP (within Container Apps environment, not exposed externally)
DatabaseSSL required (?sslmode=require in connection string)
