CI/CD Pipeline & Deployment

Deployment Flow

The platform uses GitHub Actions for automated deployment. The primary pipeline is:

StepTriggerWhat Happens
1. CIPush to main or PRLint, type-check, run tests
2. ReleaseMerge to main (semantic-release)Version bump, changelog, GitHub release, Docker image build + push to GHCR
3. DeployRelease or CI completed on main, or manual dispatchAzure Bicep deployment with new image tag

Workflow Files

FilePurpose
.github/workflows/ci.ymlContinuous integration (tests, lint, type-check)
.github/workflows/release.ymlSemantic release (versioning, changelog, Docker build)
.github/workflows/deploy.ymlPrimary: Bicep deployment to Azure Container Apps
.github/workflows/deploy-azure.ymlAlternative: Terraform-based deployment (legacy)
.github/workflows/redeploy-on-server-update.ymlAuto-redeploy when vendor MCP server images update

Manual Deployment

# Trigger a manual deploy via GitHub CLI
gh workflow run deploy.yml -f environment=production

# Or deploy directly with Azure CLI + Bicep
az deployment group create \
  --resource-group {rg} \
  --template-file azure/main.bicep \
  --parameters \
    gatewayImage=ghcr.io/wyre-technology/mcp-gateway:sha-{commit} \
    redeployTrigger=$(date +%s) \
    masterKey={key} \
    jwtSecret={key} \
    pgPassword={pass} \
    ghcrToken={token}

Rolling Back a Deployment

# List recent revisions
az containerapp revision list --name mcpgw-prod-gateway \
  --resource-group {rg} --query '[].{name:name,active:active,created:createdTime}' -o table

# Activate a previous revision
az containerapp revision activate --name mcpgw-prod-gateway \
  --resource-group {rg} --revision {previous-revision-name}

# Deactivate the bad revision
az containerapp revision deactivate --name mcpgw-prod-gateway \
  --resource-group {rg} --revision {bad-revision-name}

Database Backups & Restore

Automatic Backups

Azure PostgreSQL Flexible Server is configured with:

SettingValue
Backup retention7 days
Geo-redundant backupDisabled
Backup frequencyAutomatic (managed by Azure)

Point-in-Time Restore

# List available restore points
az postgres flexible-server show --name mcpgw-prod-pg \
  --resource-group {rg} --query backup

# Restore to a point in time (creates a NEW server)
az postgres flexible-server restore \
  --name mcpgw-prod-pg-restored \
  --resource-group {rg} \
  --source-server mcpgw-prod-pg \
  --restore-time "2026-03-30T12:00:00Z"
Important: Point-in-time restore creates a new PostgreSQL server. After verifying the restored data, update the gateway's DATABASE_URL to point to the new server and redeploy.

Manual Backup (pg_dump)

# Connect and dump
pg_dump "postgres://gatewayadmin:{password}@mcpgw-prod-pg.postgres.database.azure.com:5432/gateway?sslmode=require" \
  --format=custom --file=gateway-backup-$(date +%Y%m%d).dump

# Restore from dump
pg_restore --dbname="postgres://..." --clean --if-exists gateway-backup-20260330.dump

Certificate Renewal

TLS Certificate (Azure Managed)

The gateway uses Azure Container Apps managed certificates. These are automatically renewed by Azure before expiry. No manual intervention is needed.

Verification

# Check certificate status
az containerapp env certificate list \
  --name mcpgw-prod-env --resource-group {rg} \
  --query '[].{name:name,expiration:expirationDate,thumbprint:thumbprint}' -o table

Auth0 Signing Certificate

Auth0 manages its own signing keys. If Auth0 rotates keys, existing JWTs from Auth0 may fail verification. The gateway uses Auth0 OIDC discovery to automatically fetch current keys.

Rotating Secrets & Credentials

Rotating MASTER_KEY

WARNING: Rotating the MASTER_KEY invalidates ALL stored encrypted credentials. Every customer will need to re-enter their vendor API keys. Plan this as a scheduled maintenance event.
  1. Generate a new 32-byte hex key: openssl rand -hex 32
  2. Update the GitHub secret MASTER_KEY
  3. Redeploy via the CI/CD pipeline
  4. Notify all customers to re-enter vendor credentials

Rotating JWT_SECRET

Rotating the JWT_SECRET invalidates all active access and refresh tokens. All users must re-authenticate.
  1. Generate a new 32-byte hex key: openssl rand -hex 32
  2. Update the GitHub secret JWT_SECRET
  3. Redeploy
  4. Users will be prompted to re-authenticate on next API call

Rotating Auth0 Secrets

  1. Rotate the client secret in Auth0 Dashboard
  2. Update the Key Vault secret: 
    az keyvault secret set --vault-name mcpgw-prod-kv \
  --name auth0-client-secret --value "{new-secret}"
  3. Restart the gateway container to pick up the new Key Vault secret value

Rotating Stripe Secrets

  1. Generate new keys in Stripe Dashboard
  2. Update Key Vault: 
    az keyvault secret set --vault-name mcpgw-prod-kv \
  --name stripe-secret-key --value "{new-key}"
az keyvault secret set --vault-name mcpgw-prod-kv \
  --name stripe-webhook-secret --value "{new-secret}"
  3. Restart the gateway container

Rotating GHCR Token

  1. Generate a new GitHub PAT with read:packages scope
  2. Update GitHub secret GHCR_TOKEN
  3. Redeploy (the token is passed to the Bicep deployment for container registry auth)

Rotating PostgreSQL Password

  1. Change the password on the PostgreSQL server: 
    az postgres flexible-server update --name mcpgw-prod-pg \
  --resource-group {rg} --admin-password "{new-password}"
  2. Update the GitHub secret PG_PASSWORD
  3. Redeploy (the DATABASE_URL secret is rebuilt from the new password during Bicep deployment)

Scaling Up & Down

Gateway Horizontal Scaling

# Scale to more replicas
az containerapp update --name mcpgw-prod-gateway \
  --resource-group {rg} \
  --min-replicas 2 --max-replicas 5

# Scale back down
az containerapp update --name mcpgw-prod-gateway \
  --resource-group {rg} \
  --min-replicas 1 --max-replicas 3

Gateway Vertical Scaling

# Increase CPU/memory
az containerapp update --name mcpgw-prod-gateway \
  --resource-group {rg} \
  --cpu 1.0 --memory 2Gi

PostgreSQL Scaling

# Scale up the tier
az postgres flexible-server update --name mcpgw-prod-pg \
  --resource-group {rg} --sku-name Standard_B2ms

# Increase storage
az postgres flexible-server update --name mcpgw-prod-pg \
  --resource-group {rg} --storage-size 64

Vendor Container Scaling

Vendor containers are configured with 1 replica by default. For high-traffic vendors:

az containerapp update --name mcpgw-prod-{vendor-slug} \
  --resource-group {rg} \
  --min-replicas 1 --max-replicas 3

Updating Vendor MCP Containers

Updating a Single Vendor Image

# Pull and deploy the latest image for a specific vendor
az containerapp update --name mcpgw-prod-{vendor-slug} \
  --resource-group {rg} \
  --image ghcr.io/wyre-technology/{vendor-slug}-mcp:latest

Updating All Vendors via Bicep

Update the vendors parameter in the Bicep deployment to change images or add/remove vendors, then redeploy.

Adding a New Vendor

  1. Add the vendor to the vendors array in azure/main.bicep
  2. Add the VENDOR_URL_{SLUG} environment variable to the gateway container
  3. Configure the vendor in src/credentials/vendor-config.ts
  4. Redeploy

Disaster Recovery

Recovery Time Objectives

ScenarioRTORPOProcedure
Gateway container crash< 5 min0 (stateless)Automatic restart by Container Apps
Vendor container crash< 5 min0 (stateless)Automatic restart by Container Apps
Database corruption30-60 minUp to 5 min (continuous backup)Point-in-time restore
Region outage2-4 hoursUp to 24 hoursRedeploy to alternate region
MASTER_KEY compromised1-2 hoursN/ARotate key, notify customers

Full Region Recovery Procedure

  1. Create a new resource group in an alternate region
  2. Restore PostgreSQL from the most recent backup (if geo-redundant backup is enabled) or from a manual pg_dump
  3. Deploy the Bicep template to the new resource group: 
    az deployment group create \
  --resource-group {new-rg} \
  --template-file azure/main.bicep \
  --parameters location={new-region} env=prod \
    masterKey={key} jwtSecret={key} pgPassword={pass} ghcrToken={token}
  4. Update DNS to point to the new gateway FQDN
  5. Update Auth0 callback URL to the new domain
  6. Verify /health and /health/vendors return healthy
  7. Notify customers if credential re-entry is required
