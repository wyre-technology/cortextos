Authentication Failures

Issue: Users Cannot Log In (Auth0)

Symptoms: Redirect to Auth0 fails, callback returns error, "invalid_grant" or "unauthorized_client" errors.

Tenant-aware note (clarified 2026-06-16): Conduit-staging and conduit-prod use separate Auth0 tenants, each with its own application, client credentials, allowed-URL list, and Key Vault. The diagnosis + resolution steps below name the prod Key Vault and prod callback URL because this runbook covers production incidents; for a staging-Auth0 incident, swap in the staging tenant's Key Vault (mcpgw-staging-kv) and staging callback URL (https://staging.conduit.wyre.ai/auth/callback). See staging-environment-setup.md §4 for the per-tenant model.

Diagnosis:

  1. Check Auth0 status page: https://status.auth0.com (Auth0 status applies to both tenants — outages are global)
  2. Verify Auth0 secrets in the affected environment's Key Vault are current (the example below uses the prod Key Vault; swap for mcpgw-staging-kv on a staging incident): 
    az keyvault secret show --vault-name mcpgw-prod-kv --name auth0-domain
az keyvault secret show --vault-name mcpgw-prod-kv --name auth0-client-id
az keyvault secret show --vault-name mcpgw-prod-kv --name auth0-client-secret
  3. Verify AUTH0_CALLBACK_URL on the affected gateway matches the callback URL configured in that environment's Auth0 tenant dashboard (prod tenant for the prod gateway; staging tenant for the staging gateway)
  4. Check gateway logs for Auth0-related errors (see Log Locations)

Resolution:

  • If Auth0 secrets rotated: update the affected environment's Key Vault secrets and restart its gateway container to pick up new values (staging incidents → mcpgw-staging-kv + staging container; prod incidents → mcpgw-prod-kv + prod container)
  • If callback URL mismatch: update either the AUTH0_CALLBACK_URL env var on the affected gateway, or the application settings in that environment's Auth0 tenant (do NOT edit the other environment's tenant — they are independent and edits do not cross over)
  • If Auth0 is down: wait for Auth0 to recover; existing JWT tokens will continue to work until they expire (applies to both tenants since the outage is upstream of both)

Issue: JWT Token Expired or Invalid (401)

Symptoms: {"error":"invalid_token"} or {"error":"token_expired"} on API calls.

Diagnosis:

  1. Decode the JWT (jwt.io) and check the exp claim
  2. Verify the JWT_SECRET has not changed since the token was issued
  3. Check if the user's refresh token is still valid (30-day TTL by default)

Resolution:

  • Token expired: client should automatically refresh via the refresh token grant. If refresh also fails, user must re-authenticate
  • JWT_SECRET changed: all existing tokens are invalidated. All users must re-authenticate. This is expected if MASTER_KEY or JWT_SECRET is rotated
WARNING: If MASTER_KEY or JWT_SECRET are not set in production, the gateway generates random keys on startup. This means all tokens and encrypted credentials are invalidated on every restart. Always set these in production.

Issue: User Gets 403 Forbidden

Symptoms: User is authenticated but specific tool calls or vendor requests return 403.

Diagnosis:

  1. Check user's server access grants: GET /api/orgs/:orgId/members/:userId/server-access
  2. Check tool allowlists for the user's role: GET /api/orgs/:orgId/tool-access/:vendor/:role
  3. Check if org has hit plan limits (vendor count, rate limit):
    • Free: 3 vendors, 100 req/hour
    • Pro: unlimited vendors, 1,000 req/hour

Resolution:

  • Missing vendor access: PUT /api/orgs/:orgId/members/:userId/server-access/:vendor
  • Tool allowlist blocking: update allowlist or remove restrictions
  • Plan limit: upgrade to Pro or wait for rate limit window to reset (1 hour)

API Proxy Failures

Issue: 502/504 on MCP Tool Calls

Symptoms: Tool calls timeout or return 502 Bad Gateway / 504 Gateway Timeout.

Diagnosis:

  1. Check vendor container health: GET /health/vendors
  2. Check if the vendor container is running: 
    az containerapp show --name mcpgw-prod-{vendor-slug} \
  --resource-group {rg} --query properties.runningStatus
  3. Check vendor container logs: 
    az containerapp logs show --name mcpgw-prod-{vendor-slug} \
  --resource-group {rg} --tail 50
  4. Verify the VENDOR_URL_{SLUG} environment variable resolves correctly

Resolution:

  • Vendor container crashed: restart the container (see Container Restart)
  • Vendor API is down (upstream): check the vendor's status page. The gateway will continue to proxy requests; the vendor's API will return errors
  • Network resolution failure: verify the container app name matches the expected internal FQDN

Issue: Vendor Credentials Not Working

Symptoms: Tool calls reach the vendor but return auth errors (401/403 from the vendor API).

Diagnosis:

  1. Verify credentials are stored: check the web UI at /settings/team/connections
  2. Check credential scope resolution order: Personal → Team → Org. The most specific scope wins
  3. For OAuth vendors: check if the refresh token has expired or been revoked

Resolution:

  • API key expired/rotated: update credentials at the appropriate scope (org, team, or personal)
  • OAuth token revoked: re-authorize via the web UI "Connect" button
  • Wrong scope: verify the credential exists at the expected scope level

Database Connectivity Issues

Issue: PostgreSQL Connection Errors

Symptoms: 500 errors, ECONNREFUSED, connection terminated, connect_timeout, or too many connections in logs.

Diagnosis:

  1. Check PostgreSQL server status: 
    az postgres flexible-server show --name mcpgw-prod-pg \
  --resource-group {rg} --query state
  2. Check if connection limit is reached: 
    az postgres flexible-server parameter show \
  --server-name mcpgw-prod-pg --resource-group {rg} \
  --name max_connections
  3. Check firewall rules allow Azure services: 
    az postgres flexible-server firewall-rule list \
  --name mcpgw-prod-pg --resource-group {rg}

Resolution:

  • Server stopped: start it via the Azure Portal or CLI
  • Connection limit reached: the gateway uses a pool of 10 connections with 20s idle timeout. If exceeded, consider scaling up the PostgreSQL tier
  • Firewall blocking: ensure the "AllowAzureServices" rule exists (0.0.0.0 - 0.0.0.0)
  • Password changed: update the DATABASE_URL secret in the container app and restart

Issue: Slow Database Queries

Symptoms: High P95 response times, requests timing out.

Resolution:

  • Check PostgreSQL metrics in Azure Portal (CPU, memory, IOPS, connections)
  • The gateway runs a cleanup of request_log entries older than 90 days on startup. If the table is very large, consider running manual cleanup
  • Consider scaling up from Standard_B1ms to a higher tier if sustained load is high

Vendor Container Health Issues

Issue: Vendor Marked as DOWN

Symptoms: /health/vendors shows a vendor with "status": "down", webhook alert received.

Diagnosis:

  1. Check the lastError and consecutiveFailures fields from /health/vendors
  2. Check vendor container logs: 
    az containerapp logs show --name mcpgw-prod-{vendor-slug} \
  --resource-group {rg} --tail 100
  3. Check if the vendor container image exists and is pullable: 
    docker pull ghcr.io/wyre-technology/{vendor-slug}-mcp:latest

Resolution:

  • Container crash-looping: check logs for the root cause, then restart
  • Image pull failure: verify GHCR token is valid and the image tag exists
  • The vendor monitor will automatically detect recovery and send a "RECOVERED" alert

Container Restart Procedures

Restart the Gateway

# Create a new revision (restarts the container)
az containerapp revision restart \
  --name mcpgw-prod-gateway \
  --resource-group {rg} \
  --revision $(az containerapp revision list --name mcpgw-prod-gateway \
    --resource-group {rg} --query '[0].name' -o tsv)

Restart a Vendor Container

# Replace {vendor-slug} with the vendor name (e.g., datto-rmm)
az containerapp revision restart \
  --name mcpgw-prod-{vendor-slug} \
  --resource-group {rg} \
  --revision $(az containerapp revision list --name mcpgw-prod-{vendor-slug} \
    --resource-group {rg} --query '[0].name' -o tsv)

Force Redeploy (New Revision)

If a restart does not resolve the issue, trigger a new revision deployment:

# Gateway
az containerapp update --name mcpgw-prod-gateway \
  --resource-group {rg} \
  --set-env-vars "REDEPLOY_TRIGGER=$(date +%s)"

# Vendor container
az containerapp update --name mcpgw-prod-{vendor-slug} \
  --resource-group {rg} \
  --set-env-vars "REDEPLOY_TRIGGER=$(date +%s)"

Local Development (Docker Compose)

# Restart all services
docker compose restart

# Restart specific service
docker compose restart gateway
docker compose restart datto-rmm-mcp

# Rebuild and restart
docker compose up --build -d gateway

# View logs
docker compose logs -f gateway --tail 100

Quick Log Access via Azure CLI

# Real-time gateway logs
az containerapp logs show --name mcpgw-prod-gateway \
  --resource-group {rg} --follow --tail 50

# Vendor container logs
az containerapp logs show --name mcpgw-prod-datto-rmm \
  --resource-group {rg} --tail 100

# Log Analytics query (last hour of errors)
az monitor log-analytics query --workspace mcpgw-prod-logs \
  --analytics-query "ContainerAppConsoleLogs_CL | where ContainerAppName_s == 'mcpgw-prod-gateway' | where Log_s has 'error' | order by TimeGenerated desc | take 20" \
  --timespan PT1H
