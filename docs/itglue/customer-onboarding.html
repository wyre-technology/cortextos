Onboarding Overview

Onboarding a new MSP customer onto the MCP Gateway involves creating an organization, connecting their vendor API credentials, configuring team access, and setting up their MCP clients (Claude Desktop / Claude Code). The process typically takes 30-60 minutes for a straightforward deployment.

Phase 1: Discovery

1 Identify Customer Vendor Stack

Work with the MSP to inventory which vendors they use. The gateway is vendor-agnostic; vendors are enabled per-customer.

Phase 2: Organization Setup

3 Create the Organization

The MSP owner navigates to the gateway web UI at /settings and creates a new organization, or use the API:

POST /api/orgs
{
  "name": "Acme MSP",
  "invite_code": "<alpha-invite-code>"   // Optional: grants Pro plan immediately
}
An alpha invite code immediately upgrades the org to Pro. Without one, the org starts on the Free plan (3 vendor connections, 100 req/hour, 1 member).

4 Upgrade to Pro Plan

If not using an invite code, upgrade via one of these methods:

MethodAPI Call
Stripe checkoutPOST /api/billing/checkout { "org_id": "..." }
Redeem invite code laterPOST /api/orgs/:orgId/redeem-code { "code": "..." }
Admin direct set (managed contracts)POST /api/admin/orgs/:orgId/plan { "plan": "pro" } (requires X-Admin-Api-Key header)

Pro Plan Unlocks

  • Unlimited vendor connections
  • 1,000 requests/hour/vendor
  • Team features (members, roles, invitations)
  • Tool allowlists (per-vendor, per-role RBAC)
  • Audit log access and CSV export
  • Log shipping to SIEM (Loki, Graylog, LogScale)
  • Service clients for M2M automation
  • Prompt/argument capture in audit logs

Phase 3: Vendor Credentials

5 Connect Vendor API Credentials

For each vendor the MSP uses, configure credentials at the org level.

API-Key Vendors (most vendors)

Navigate to /settings/team/connections in the web UI and click "Connect" next to the vendor. Enter the required API keys/secrets. Alternatively, use the API:

POST /api/orgs/:orgId/credentials/:vendor
{
  "apiKey": "...",
  "apiSecret": "..."
}

OAuth Vendors (Xero, QuickBooks, HubSpot, M365)

Click "Connect" in the web UI and complete the OAuth flow. The gateway handles token exchange and stores refresh tokens for automatic renewal.

OAuth vendor app credentials (e.g., XERO_CLIENT_ID, MICROSOFT_CLIENT_ID) are configured at the gateway level via environment variables, not per-customer. These are the Wyre Technology app registrations that customers authorize.

6 Configure Server Access

By default, new members get access based on the org's defaultServerAccess setting. To control per-member vendor access:

# Set which vendors a member can access
PUT /api/orgs/:orgId/members/:userId/server-access
{
  "vendors": ["datto-rmm", "itglue", "autotask"]
}

# Grant individual vendor access
PUT /api/orgs/:orgId/members/:userId/server-access/datto-rmm

Phase 4: Teams & RBAC

7 Set Up Teams (Optional)

For MSPs with distinct groups (e.g., "Tier 1 Support", "Security Team"):

POST /api/orgs/:orgId/teams
{ "name": "Tier 1 Support" }

Teams can have their own vendor credentials (separate API keys) and their own vendor access grants:

PUT /api/orgs/:orgId/teams/:teamId/server-access/datto-rmm

8 Configure Tool Allowlists (Optional)

Restrict which tools a role can access per vendor:

PUT /api/orgs/:orgId/tool-access/datto-rmm/member
{
  "tools": ["datto_list_devices", "datto_list_sites", "datto_list_alerts"]
}

Owners always have full access. Setting no allowlist means all tools are available.

9 Invite Team Members

POST /api/orgs/:orgId/invitations

Returns an invitation URL like https://gateway.example.com/invite/<token>. Share with team members. When they visit the link and sign in via Auth0, they join the org as a member.

To promote a member to admin:

PATCH /api/orgs/:orgId/members/:userId/role
{ "role": "admin" }

10 Create Service Clients (Optional)

For M2M automation (AI agents, scripts, integrations):

POST /api/orgs/:orgId/service-clients
{
  "name": "Nightly Ticket Sync",
  "expires_in_days": 90
}

Returns client_id and client_secret (shown once). The service client authenticates via client_credentials grant and receives a JWT scoped to the org.

Phase 5: Client Configuration

11 Configure MCP Clients

Claude Desktop

Add to claude_desktop_config.json:

{
  "mcpServers": {
    "wyre": {
      "url": "https://gateway.example.com/v1/mcp"
    }
  }
}

Claude Code

claude mcp add wyre https://gateway.example.com/v1/mcp

On first use, the OAuth flow opens a browser for authentication. After sign-in and credential connection, the JWT is cached and refreshed automatically.

CLI Wrapper (REST)

For scripts and automation that do not need the full MCP JSON-RPC protocol:

POST /api/cli/:vendor/:tool
Authorization: Bearer <jwt>
Content-Type: application/json

{ "site_id": "12345" }

12 Configure Log Shipping (Optional)

POST /api/orgs/:orgId/log-shipping
{

Go-Live Verification Checklist

  • Organization created and on correct plan (Free or Pro)
  • All required vendor credentials connected and validated (test via /health/vendors)
  • Server access grants configured for each member
  • Tool allowlists set for restricted roles (if applicable)
  • Team members invited and have accepted invitations
  • Claude Desktop or Claude Code configured with the gateway MCP server URL
  • Service clients created for any automation workflows (if applicable)
  • Test tool call made and verified in audit log (GET /api/audit)
  • Log shipping configured and tested (if applicable)
  • MSP owner has verified Stripe billing portal access (if using Stripe)
  • White-label branding verified (if configured)
  • Customer has received documentation on how to manage their org via the web UI

Rollback Procedure

If onboarding fails or the customer needs to be removed:

Step 1: Disable Vendor Access

# Remove all vendor access for the org's members
PUT /api/orgs/:orgId/members/:userId/server-access
{ "vendors": [] }

Step 2: Revoke Service Clients

# Delete any service clients
DELETE /api/orgs/:orgId/service-clients/:clientId

Step 3: Remove Credentials

# Delete vendor credentials (they are encrypted; deletion removes them from the database)
DELETE /api/orgs/:orgId/credentials/:vendor

Step 4: Remove Members

# Remove all non-owner members
DELETE /api/orgs/:orgId/members/:userId

Step 5: Delete Organization

# This cascades to all related data (credentials, audit logs, log shipping, teams)
DELETE /api/orgs/:orgId

Step 6: Cancel Billing

If Stripe is configured, cancel the subscription via the Stripe dashboard or API. The billing portal is accessible at POST /api/billing/portal.

Important: Organization deletion is permanent. All encrypted credentials, audit logs, and team data will be removed. Export audit logs before deletion if needed for compliance.
