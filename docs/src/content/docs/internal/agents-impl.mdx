## Agent model

An agent is an MCP prompt registered against Conduit's prompt server.
Each agent carries:

- A **name** (kebab-case, unique across the plugin).
- A **plugin binding** — the vendor integration (or `portfolio`) that
  owns it.
- A **trigger phrase set** — human-language triggers surfaced in
  Claude's prompt picker search.
- A **system prompt** — embeds the domain knowledge and output
  contract.
- A **tool manifest** — declares the vendor tools the agent expects
  access to so the session can pre-validate authorization.
- An **output schema** — a shape the model is instructed to produce;
  used for downstream formatters (PSA ticket creation, PDF export).

Registration lives under `src/agents/<plugin>/<agent-name>.ts` in
the Conduit repo. The agent index (`src/agents/index.ts`) assembles
the final prompt list surfaced on the MCP prompts endpoint.

## Resolution at runtime

1. Claude Desktop lists prompts via the MCP `prompts/list` endpoint.
   Conduit filters the list to agents whose tool manifest is fully
   satisfied by the caller's tool allowlist — agents the user cannot
   run are hidden.
2. The user picks an agent. Claude Desktop sends `prompts/get` with
   the agent name and any arguments.
3. Conduit returns the system prompt with arguments interpolated.
4. Claude begins the session and calls vendor tools through the MCP
   transport as normal. Every call is audited under the user's
   identity.

## Authoring conventions

- **Naming.** Kebab-case, verb-first where possible
  (`ticket-dispatcher`, not `dispatch-tickets`). Plugin-owned agents
  do not prefix the plugin name; portfolio agents stand alone.
- **Output contract first.** Write the desired output section
  headings before the system prompt body. The prompt exists to
  produce the output, not the other way around.
- **Vendor-specific knowledge goes in the prompt, not in tool
  responses.** Example: Autotask Priority 4 = Critical is encoded in
  the ticket-dispatcher system prompt, not in a tool wrapper.
- **Trigger phrases cover intent, not jargon.** "fleet health check"
  and "device audit" are good; "rmm-health-auditor" is redundant.
- **Example queries are required.** At least three per agent. They
  calibrate the model and help technicians discover the agent in
  Claude's search.
- **Never include Wyre-internal branding.** Output that a customer
  may see must use the MSP's configured brand tokens, never "Wyre
  Gateway".

## Authorization surfacing

The MCP `prompts/list` endpoint filters the returned list to agents
the caller is authorized to run in full. The filter looks at:

- The caller's role (`admin` inherits all tools; `user` is bounded
  by the allowlist).
- The customer's connected vendor integrations.
- The customer's tool allowlists per vendor.

If any required tool in an agent's manifest is not reachable, the
agent is omitted. This prevents a prompt picker full of agents the
user cannot actually run.

## Custom-agent registration

MSPs with custom agents register them at MSP-org scope. The
registration path is:

```
POST /api/msps/<msp-slug>/agents
```

with a JSON body matching the agent model. The endpoint is admin-only
and requires the `agents:write` permission. Custom agents are scoped
to the MSP and visible to every customer under it.

Custom agents follow the same authoring conventions as built-ins;
enforcement is linted at registration time.
